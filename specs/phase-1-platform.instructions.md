# Phase 1 — Learning Platform

## Objective

A complete platform that evaluates the user's English CEFR level through a 3-step onboarding, generates a deterministic curriculum-driven study plan, presents interactive lessons generated by an LLM within strict curriculum boundaries, provides SM-2 spaced repetition flashcards, and offers an AI tutor with SSE streaming — all connected through a Next.js frontend with JWT+Redis authentication.

## Milestone 2 — Backend Core

### Configuration (`app/core/config.py`)

Central `Settings` class using pydantic-settings, reading from `.env`. Covers:

- **Database**: `DATABASE_URL` (asyncpg connection string)
- **Redis**: `REDIS_URL`
- **JWT**: `SECRET_KEY`, `ACCESS_TOKEN_EXPIRE_MINUTES` (15), `REFRESH_TOKEN_EXPIRE_DAYS` (30)
- **Registration**: `ALLOW_REGISTRATION`, `FIRST_USER_IS_ADMIN`
- **LLM**: `LLM_PROVIDER` (ollama/openai/anthropic/deepseek) with per-provider URLs, models, and API keys
- **TTS/STT**: `TTS_ENABLED`, `STT_ENABLED` (both default false), `STT_MODEL`, `STT_ENGINE`
- **Rate limiting**: `RATE_LIMIT_ENABLED`, `RATE_LIMIT_STORAGE` (memory/redis)
- **CORS**: `CORS_ORIGINS`
- **Logging**: `LOG_LEVEL` (default INFO)
- **Cookie**: `COOKIE_SECURE` (for httpOnly refresh token)

### Database (`app/core/database.py`)

- Async engine via `create_async_engine` with asyncpg
- Async session factory (`async_sessionmaker`) with `expire_on_commit=False`
- `DeclarativeBase` for all SQLAlchemy models
- `get_db()` async generator for FastAPI dependency injection

### Security (`app/core/security.py`)

- `hash_password(password)` — bcrypt via passlib
- `verify_password(plain, hashed)` — bcrypt comparison
- `dummy_verify()` — constant-time no-op for anti-user-enumeration (login endpoint calls this for non-existent users so timing doesn't leak existence)
- `create_access_token(user_id, role)` — JWT HS256, 15 min expiry, payload: `{sub, role, exp}`
- `decode_access_token(token)` — JWT decode, raises on expiry/invalid
- `create_refresh_token()` — 64-char URL-safe random string (opaque, not JWT)

### Dependencies (`app/core/deps.py`)

- `oauth2_scheme` — OAuth2PasswordBearer at `/api/auth/login`
- `get_current_user(token)` — decodes JWT, fetches active user from DB, returns User model
- `require_admin(current_user)` — checks `role == "admin"`, returns 403 otherwise

### Auth endpoints (`app/routers/auth.py`)

- **`POST /register`** — Validates ALLOW_REGISTRATION gate, creates user (auto-admin for first), returns user info
- **`POST /login`** — Rate-limited (10/min). Verifies credentials. Returns access_token in body + sets refresh_token httpOnly cookie. Stores `refresh:{token}` → user_id in Redis with 30-day TTL. Uses dummy_verify for non-existent users
- **`POST /refresh`** — Reads refresh_token from cookie, validates against Redis, deletes old token (rotation), creates new one, returns new access_token
- **`POST /logout`** — Deletes refresh_token from Redis, clears cookie
- **`GET /me`** — Returns authenticated user profile
- **`PATCH /me`** — Updates display_name, email, password, english_variant, conversation_max_duration, conversation_inactivity_timeout

### Admin endpoints (`app/routers/admin.py`)

All require `role="admin"` via `require_admin` dependency.

| Endpoint             | Behavior                                                                                   |
| -------------------- | ------------------------------------------------------------------------------------------ |
| `GET /users`         | Lists users with pagination, search, subscription, role, and active-status filters         |
| `POST /users`        | Creates user directly (bypasses ALLOW_REGISTRATION)                                        |
| `GET /users/{id}`    | Single user detail                                                                         |
| `PATCH /users/{id}`  | Edit role, is_active, display_name                                                         |
| `DELETE /users/{id}` | Cascading delete of all user data                                                          |
| `POST /invite`       | Generates single-use invite token (48h Redis TTL). Returns URL: `/register?invite=<token>` |

### LLM Adapter (`app/services/llm_adapter.py`)

Singleton providing provider-agnostic LLM access. Supports four providers:

| Provider  | Client library                                          | Max tokens |
| --------- | ------------------------------------------------------- | ---------- |
| ollama    | AsyncOpenAI (openai SDK via openai-compatible endpoint) | 8192       |
| openai    | AsyncOpenAI                                             | 128K       |
| deepseek  | AsyncOpenAI (openai-compatible endpoint)                | 128K       |
| anthropic | AsyncAnthropic (anthropic SDK, separate code path)      | 200K       |

Key design: Ollama, OpenAI, and DeepSeek all use the same `AsyncOpenAI` client with different `base_url` values — the rest of the application never knows which provider is active.

Two core methods:

- `chat(messages, stream=False)` → returns string or async generator of chunks
- `structured_output(messages, schema)` → returns validated Pydantic model (via JSON mode + retry with correction on parse failure)

Error handling: 2 automatic retries with exponential backoff, 60 s timeout. Custom exception hierarchy: `LLMError` (base), `LLMTimeoutError`, `LLMUnavailableError`, `LLMResponseError`, `LLMContextOverflowError`.

For Anthropic, the `system` message is extracted and passed as a separate parameter (Anthropic API convention). All other providers receive it as a regular message with `role: "system"`.

## Milestone 3 — Assessment (CEFR Level Evaluation)

### 3-step onboarding flow

The assessment is not a direct quiz — it's a guided onboarding with three steps:

```
Step 1: BeginnerGate
  "Have you studied English before?"
  → "I am a complete beginner" → skip quiz → CEFR = A1, create plan directly
  → "Yes, I have some experience" → continue to Step 2

Step 2: Adaptive quiz (12 questions from static bank)
  Algorithm: start at A2 difficulty.
    - 2 correct in a row → move up one level
    - 2 wrong in a row → move down one level
    - Stop after 12 questions or when confidence threshold reached
  Questions fetched from GET /api/assessment/bank (~100 curated questions per language)
  Skills tracked: grammar, vocabulary, reading

Step 3: Duration & goals
  Duration: Intensive (4 weeks), Standard (8 w), Relaxed (12 w, default), Very Relaxed (16 w)
  Goals (multi-select): Grammar, Vocabulary, Reading, Writing, Conversation
  → Persists results, creates StudyPlan
```

### Static question bank (`backend/app/data/{lang}/assessment_bank.py`)

Approximately 100 curated questions per language spanning A1 to C2. Each question has:

- `id` (e.g. `"g-a1-001"`)
- `skill` (grammar, vocabulary, or reading)
- `difficulty` (CEFR level)
- `question` text
- `options` (exactly 4 strings)
- `correct` answer (must match one option)
- Optional `grammar_slug` (links result to grammar reference)

Questions are never generated by LLM — they are curated at build time for reliability and consistency.

### Deterministic CEFR evaluation (backend service)

The LLM is **not** used to evaluate quiz answers. A deterministic algorithm (`evaluate_adaptive_quiz`) groups answers by difficulty level and finds the highest level where the user has at least 2 questions and >= 60% correct. This provides a reliable, reproducible CEFR placement without LLM variability.

The algorithm also computes per-skill scores (grammar, vocabulary, reading) and identifies strengths (>= 0.65) and weaknesses (< 0.45) for the skill profile.

### Free-write evaluation (LLM)

An optional free-write question at the end of the quiz is evaluated by the LLM. The prompt asks the model to assess vocabulary range, grammar accuracy, and coherence, and may adjust the preliminary CEFR level by ±1 step. Returns a JSON with adjusted_level, writing_score, analysis, strengths, and weaknesses.

## Milestone 4 — Study Plan Generator

### Curriculum-driven, not free-form LLM

The study plan does **not** ask the LLM to invent a schedule. Instead, it reads the canonical curriculum from `app/data/curriculum.py` (543 lines, 24 units across A1-C2 with grammar points, vocabulary sets, lesson types, and competency checklists per unit).

The `distribute_units()` function maps curriculum units onto lesson slots based on the chosen duration and intensity. For example:

- A1 level with 12 weeks × 4 days/week = 48 lesson-days
- 8 A1 curriculum units → ~6 lesson-days per unit
- Each unit cycles through lesson types: grammar → vocabulary → reading → writing → review
- The **last slot** is always reserved for the end-of-level completion test

The plan skeleton is fully deterministic — no LLM call during plan generation. The LLM is invoked only when a lesson is opened for the first time (to generate its actual content). The generated plan is stored as JSON in the StudyPlan model and rendered as a visual roadmap on the frontend.

### End-of-level test

When the user completes all units in a level, an end-of-level test becomes available. The LLM generates 20 questions covering only grammar points and vocabulary sets studied during that level (constrained by the curriculum, not free-form).

Scoring thresholds:

| Score       | Recommendation | Effect                                     |
| ----------- | -------------- | ------------------------------------------ |
| >= 0.75     | `advance`      | Unlock next CEFR level                     |
| 0.55 – 0.74 | `extend`       | 4-week extension on weak units recommended |
| < 0.55      | `repeat`       | Full level repetition recommended          |

## Milestone 5 — Lesson Generator

### Content generation (`app/services/lesson_generator.py`)

Lessons are generated by the LLM on first access (lazy generation). The prompt is highly constrained:

- **Grammar constrained**: only uses grammar slugs from a validated set of 24 slugs (`VALID_GRAMMAR_SLUGS`)
- **CEFR level constraints**: A1 limits sentence length to 8 words and vocabulary to high-frequency items; A2 allows up to 12 words and introduces comparatives/modal verbs; B1 removes sentence length limits and introduces perfect tenses/conditionals
- **English variant awareness**: respects `user.english_variant` (american/british) for spelling and grammar conventions
- **Exercise mix**: generates 3-5 exercises with at least one free_write type. Exercise types: `multiple_choice`, `fill_blank`, `free_write`
- **Vocabulary**: 4-6 items from the unit's vocabulary set with definitions and examples
- **Grammar refs**: 1-3 slugs from the grammar reference for cross-linking

### Lesson content schema (JSON)

Each lesson stores its content as structured JSON with:

- `lesson_type`, `title`, `cefr_level`
- `explanation`: markdown text, key points list, worked examples
- `exercises`: array of exercises, each with type, question, options/correct_answer, and evaluation criteria (for free_write)
- `vocabulary`: word, definition, example items
- `grammar_refs`: grammar slugs for cross-linking to reference pages

### Exercise evaluation

**Multiple choice**: deterministic comparison against correct_answer.

**Free writing**: evaluated by LLM. The prompt receives the exercise prompt, evaluation criteria, and student's answer. Returns a score (0.0–1.0), feedback text, and specific corrections with explanations.

**Pronunciation** (Phase 2): audio recorded by browser, transcribed via STT service, then LLM evaluates the transcription against the expected text. Returns score and feedback.

### Lesson completion

When a lesson is completed:

1. All exercise scores are persisted
2. `progress_service.update_daily_progress` is called: adds XP (20 per lesson completed, 5 per correct exercise, 1 per wrong), updates streak
3. `progress_service.upsert_unit_competency` is called for each competency in the unit (exponential moving average, mastered at >= 0.80)
4. Lesson is marked `is_completed = True`

## Milestone 6 — Flashcards with SM-2

### SM-2 algorithm (`app/services/flashcard_sm2.py`)

Full SuperMemo-2 spaced repetition algorithm on the `Flashcard` model:

- **Quality scale**: 0 (complete blackout), 1 (wrong), 2 (wrong but remembered answer), 3 (correct with difficulty), 4 (correct), 5 (perfect)
- **Quality < 3**: resets repetitions to 0, sets interval to 1 day
- **Quality >= 3**: intervals follow SM-2 progression — first review: 1 day, second: 6 days, subsequent: `interval × ease_factor`. Repetitions increment
- **Ease factor**: adjusted by `EF = EF + 0.1 - (5 - quality) × (0.08 + (5 - quality) × 0.02)`, clamped to minimum 1.3
- **Next review**: `today + interval_days`

Cards due for review are queried by `next_review <= today`, ordered by increasing ease_factor (harder cards first).

### LLM flashcard generation

Users can request the LLM to generate N flashcards on a topic. The prompt includes:

- Count and topic
- Student's CEFR level
- Student's native language (for the translation field)

Returns structured JSON with flashcards containing: word, definition (in English), example_sentence, and translation (in user's native language).

### Frontend flashcard modes

- **Standard mode**: shows English word, user recalls meaning, flips for definition/translation/example, rates quality 0-5
- **Speaking mode** (Phase 2): shows English word, user pronounces it aloud, STT transcribes, comparison against expected pronunciation

## Milestone 7 — Chat with AI Tutor

### Tutor system prompt

The chat tutor uses a carefully crafted system prompt injected with:

- Student's CEFR level and native language
- Current topics from the active study plan unit
- Recent progress stats (streak, XP, skill scores)
- Recent mistakes to address

Key behavioral rules:

- Adapt vocabulary and sentence complexity to the student's level
- Gently correct grammar mistakes when they occur
- Brief explanations in the student's native language are allowed for clarity, but the main conversation stays in English
- Keep responses concise (2-4 sentences) unless explaining grammar
- **Never use emojis or emoticons** — TTS reads them aloud as text

### SSE streaming

The chat endpoint uses Server-Sent Events via FastAPI's `StreamingResponse`. The LLM adapter's streaming mode (`chat(messages, stream=True)`) yields content chunks. Each chunk is framed as `data: <content>\n\n`. The stream terminates with `data: [DONE]\n\n`.

The frontend reads the SSE stream using the Fetch API's `ReadableStream` reader, decoding chunks and updating the chat UI in real time.

### Conversation management

Text chat messages are grouped into conversations:

- Users can create, list, and delete conversations
- Each conversation contains ordered chat history messages (user/assistant roles)
- Conversations cascade-delete their messages
- History is persisted in PostgreSQL (unlike the Phase 3 voice conversation, which is in-memory only)

### Frontend SSE proxy

To avoid Next.js buffering the SSE stream (since Next.js API rewrites buffer responses), the chat uses a dedicated Next.js Route Handler at `src/app/api/chat/route.ts` that acts as a transparent proxy to the backend.

### State management (Zustand stores)

Four stores, all in JS memory (no persistence except theme to localStorage):

| Store         | Key state                                                                      | Purpose                                           |
| ------------- | ------------------------------------------------------------------------------ | ------------------------------------------------- |
| `auth.ts`     | `accessToken`, `User` object                                                   | In-memory JWT; user profile; logout action        |
| `progress.ts` | streak, xp, skills, todayLessons, currentUnitId, unitProgress, levelTestResult | Learning progress and curriculum state            |
| `theme.ts`    | `theme` ('system'/'dark'/'light')                                              | UI theme, persisted to `fl-theme` in localStorage |
| `loading.ts`  | `count` (global loading counter)                                               | When > 0, top loading bar appears                 |

### API client with auto-refresh (`lib/api.ts`)

The `apiFetch` wrapper:

1. Attaches `Authorization: Bearer <token>` header from auth store
2. On 401 response: calls `POST /api/auth/refresh` (using httpOnly cookie), stores new access_token, retries the original request
3. Uses a mutex to prevent concurrent refresh calls
4. On refresh failure: clears auth state, redirects to `/login`
5. Increments/decrements global loading counter for the loading bar

### Route protection (middleware)

Next.js middleware at `src/middleware.ts`:

- Public routes: `/login`, `/register`
- Protected routes: everything else
- Checks for `refresh_token` cookie presence
- Redirects to `/login` if no cookie on a protected route
- Also handles locale detection via `next-intl`

### Visual roadmap (`/plan`)

The roadmap renders curriculum units as a timeline. Each unit card shows:

| State      | Visual                             | Condition                           |
| ---------- | ---------------------------------- | ----------------------------------- |
| Completed  | Green, shows score %               | All lessons in unit completed       |
| Active     | Accent color, shows "Day X / Y"    | Prerequisite met, currently working |
| Unlocked   | Grey with border, clickable        | Prerequisite met but not started    |
| Locked     | Grey with lock icon, not clickable | Prerequisite not met                |
| Level test | Gold, at end of roadmap            | All units completed in the level    |

Clicking a unit opens a drawer (`UnitDrawer`) showing the unit's grammar points, lesson list with completion status, and start buttons for available lessons.

### Loading bar

A global animated loading bar (`LoadingBar` component) appears at the top of the page whenever the Zustand loading store count > 0. All API calls through `apiFetch` automatically increment/decrement this counter, providing visual feedback for every network request.

### Theme system

The `ThemeProvider` component applies a `data-theme` attribute to the HTML element based on the Zustand theme store. Three modes:

- **System**: follows OS `prefers-color-scheme` in real time (default)
- **Dark**: always dark
- **Light**: always light
